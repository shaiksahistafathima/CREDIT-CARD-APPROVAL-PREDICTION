System Documentation & Technical Specifications

Comprehensive overview of the datasets, data pre-processing logic, weak supervision rules, and data visualization findings.

1. Executive Overview

Banks receive millions of credit card applications monthly. Due to the high risk of defaults, manual verification is slow and prone to errors. This system automates the screening process using a machine learning model trained on historical demographic profiles and monthly account repayment behaviors.

2. Dataset Schema & Data Dictionary

The model utilizes two main inputs merged by a unique key (ID):

File Source Feature Name Data Type Description / Value Range
Application Profile CODE_GENDER Categorical Gender of the applicant (M = Male, F = Female)
Application Profile AMT_INCOME_TOTAL Numerical Total annual income of the applicant
Application Profile DAYS_EMPLOYED Numerical Employment length in days (Negative values; 365243 = Unemployed)
Application Profile DAYS_BIRTH Numerical Age in days (Negative values)
Application Profile OCCUPATION_TYPE Categorical Job role category (e.g. Managers, Laborers, Unknown)
Credit Ledger STATUS Categorical Monthly repayment status (X=No loan, C=Closed, 0=On-time, 1-5=Months Late)

3. Feature Engineering & Preprocessing Pipeline

  • Missing Value Imputation: The OCCUPATION_TYPE column contains 30.6% missing values (NaN). We map this missingness to a custom string category "Unknown" to keep the record without losing other valuable profile metrics.
  • Employment Translation: Raw employment is stored in days. We calculate YEARS_EMPLOYED by dividing by 365 and bounding it to ≥ 0. For unemployed entries (identified by code 365243), we set the value to 0.
  • Credit History Aggregation: Transaction history contains multiple monthly logs per client. We group them by ID to engineer fixed-size metrics:
    • STATUS_MAX: Worst credit status observed (highest delay code).
    • STATUS_MEAN: Average numerical credit delay code.
    • STATUS_TREND: Direction of monthly payment behaviors (improving or declining).
    • NUM_LATE_MONTHS: Number of months where payment was delayed.

4. Label Generation Heuristics (Target)

Since real-world credit ledger datasets are unlabeled (lacking a clean approved/denied verdict column), a Weak Supervision Heuristic Rule was defined to generate targets ($y$):

An applicant receives 1 point for each of the following rules they meet:
  • Income Check: Annual income is ≥ $120,000
  • Employment Check: Employment tenure is ≥ 2 years
  • Credit History Check: Payment status metrics indicate clean repayment history (no delays and ≤ 1 late month total)
  • Family Size Check: ≤ 2 children and ≤ 4 total family members

Verdict Label = Approved (1) if they score ≥ 3 points; else Denied (0).
